= Arquitectura de la solución
:hardbreaks:
:allow-uri-read: 
:nofooter: 
:icons: font
:linkattrs: 
:imagesdir: ../media/


[role="lead"]
El siguiente diagrama de arquitectura ilustra una implementación de base de datos Oracle de alta disponibilidad en una instancia de AWS EC2 con el servicio de almacenamiento FSx.  Se puede configurar un esquema de implementación similar, pero con el servidor en espera en una región diferente, para la recuperación ante desastres.

Dentro del entorno, la instancia de cómputo de Oracle se implementa a través de una consola de instancia de AWS EC2.  Hay varios tipos de instancias EC2 disponibles desde la consola.  NetApp recomienda implementar un tipo de instancia EC2 orientada a bases de datos, como una imagen Ami m5 con RedHat Enterprise Linux 8 y hasta 10 Gps de ancho de banda de red.

Por otro lado, el almacenamiento de la base de datos de Oracle en volúmenes FSx se implementa con la consola o CLI de AWS FSx.  Posteriormente, los volúmenes binarios, de datos o de registro de Oracle se presentan y se montan en un host Linux de instancia EC2.  Cada volumen de datos o registro puede tener múltiples LUN asignados según el protocolo de almacenamiento subyacente empleado.

image:aws-ora-fsx-ec2-arch.png["Esta imagen muestra un diagrama de arquitectura de muestra que incluye el clúster de HA principal, el clúster de HA en espera, los nodos de administración y los nodos de conexión relacionados."]

Un clúster de almacenamiento FSx está diseñado con doble redundancia, de modo que los clústeres de almacenamiento principal y en espera se implementan en dos zonas de disponibilidad diferentes.  Los volúmenes de base de datos se replican desde un clúster FSx principal a un clúster FSx en espera en un intervalo configurable por el usuario para todos los volúmenes binarios, de datos y de registro de Oracle.

Este entorno Oracle de alta disponibilidad se administra con un nodo controlador Ansible y un servidor de respaldo SnapCenter y una herramienta de interfaz de usuario.  La instalación, configuración y replicación de Oracle se automatizan mediante kits de herramientas basados en playbooks de Ansible.  Cualquier actualización del sistema operativo del kernel de la instancia Oracle EC2 o la aplicación de parches de Oracle se puede ejecutar en paralelo para mantener el servidor principal y el servidor en espera sincronizados.  De hecho, la configuración de automatización inicial se puede ampliar fácilmente para realizar algunas tareas diarias repetitivas de Oracle si es necesario.

SnapCenter proporciona flujos de trabajo para la recuperación de bases de datos Oracle en un punto determinado del tiempo o para la clonación de bases de datos en las zonas principales o en espera, si es necesario.  A través de la interfaz de usuario de SnapCenter , puede configurar la copia de seguridad y la replicación de la base de datos de Oracle en el almacenamiento FSx en espera para lograr alta disponibilidad o recuperación ante desastres en función de sus objetivos de RTO o RPO.

La solución proporciona un proceso alternativo que ofrece capacidades similares a las disponibles en la implementación de Oracle RAC y Data Guard.
